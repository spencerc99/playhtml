Add an identity, authentication, and permissions system (**beta** — works end to end, but the API and behavior may change as it's refined based on feedback). Anonymous users now get a real ECDSA P-256 keypair (non-extractable private key in IndexedDB) whose public key becomes their stable pid, in the same `pk_…` format the "we were online" extension uses — extension identities take precedence via the existing `playhtml:configure-identity` bridge. A new `permissions` init option declares roles (public-key lists or client-side condition functions) and rules gating element writes; checks are synchronous via `playhtml.can()` / the new `useCan()` React hook, `usePlayerIdentity()` now reports `verified` and `roles`, and denied writes fire a `permissiondenied` event instead of writing. When a site publishes `/.well-known/playhtml.json`, the partykit server enforces those rules for real: connections prove key ownership through a challenge–response handshake over the existing WebSocket (one signature per connection, session-token resume on reconnect, zero per-message crypto), and writes to gated elements are server-mediated — including keyed-collection entry rules (`create`/`update`/`delete` with a built-in `creator` role, server-stamped `createdBy`). Sites that don't opt in pay no cost. Note: existing anonymous users' pids change once when the fake random-hex publicKey is upgraded to a real key.

Ergonomics: rules are declared as an `elements` map sharing the `permissions` attribute's mini-language (`{ "site-title": "write:admin" }`) in init config and the well-known file alike; raw `pk_…` keys work anywhere a role name does (single-owner sites need no `roles` config); publishing the well-known file alone is enough — the server pushes rules to the browser so `can()`/`useCan()` work with zero init config, and playhtml warns about client rules the server can't enforce (CSS selectors, drifted config). `withSharedState`/`CanPlayElement` accept a `permissions` prop and pass an element-scoped `can()` to the render callback; `can()` accepts `{ entry }` and `playhtml.me.owns(entry)` checks entry ownership; `playhtml.verify()` returns a `Promise<boolean>`; `playhtml.onIdentityChange(cb)` subscribes to identity/permission changes; room enforcement state moved from `me.enforced` to `playhtml.permissionsEnforced`. Roles can also be **earned by showing up**: `{ "days": N }` role definitions grant the role to verified identities the server has seen on N distinct days in the room (counted server-side per key, once per day), with totals reported back in `playhtml.me.counters` (`days`, plus `sessions` for total verified handshakes) — enabling natural trust ladders like "returning visitors may write, regulars may moderate".
